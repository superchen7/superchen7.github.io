Transaction: n. 事务; n. 交易。例如，一次下单（设计订单的保存、库存修改等业务和 spring 的多个事务执行）。

QPS（重要）

QPS(Query per Second）:系统每秒处理查询次数，单位是次/秒。

对于互联网业务中，如果某些业务有且仅有一个请求连接，那么 TPS=QPS=HPS，一般情况下用 TPS 来衡量整个业务流程，用 QPS 来衡量接口查询次数，用 HPS 来表示对服务器单击请求。

无论TPS、QPS、HPS,此指标是衡量系统处理能力非常重要的指标，越大越好，根据经验，一般情况下：

  • 金融行业:1000TPS~5000OTPS，不包括互联网化的活动（例如，秒杀活动）
  • 保险行业:100TPS~10000OTPS，不包括互联网化的活动
  • 制造行业:10TPS~5000TPS
  • 互联网电子商务:10,000 TPS~1,000,000 TPS
  • 互联网中型网站:1,000 TPS~50,000 TPS
  • 互联网小型网站:500 TPS~10,000 TPS

90%响应时间（90%Response Time）是指所有用户的响应时间进行排序，第90%的响应时间。

我们的目标是使 90% 的请求都符合需求即可。

3.3.优化步骤（重点）

1.使用 jmeter 压测 nginx、网关、直接访问服务、通过链路访问服务

2.测 nginx 时使用 docker stats 查看 CPU 占用+内存使用量，调整 CPU 性能

测试地址：本机 nginx 地址是 127.0.0.1:80

可以注意到，测试前后 MEM 变动不大 -> 不吃内存；但是 CPU 上升 -> 吃 CPU

3.测网关时，打开 jvisualvm 查看内存、CPU使用量，调整CPU、内存

我们设定的网关的地址：127.0.0.1:88，此时响应内容可能都是 404，但是不影响。

增加 jvm 内存后重新测试：-Xmx512m

6.总结：

1）中间件越多，性能损失越大，损失在网络交互

2）命中率不低于95%

3）锁等待次数越低、时间越短越好

4）优化业务：

  • DB（MYSQL优化，加索引，例如，parent_cid pms_category 表）在 navicat 中管理索引，右键添加

  • 模板渲染速度（thymeleaf 开启缓存） spring.thymeleaf.cache = true

  • 静态资源（把静态资源放到 nginx 里面）

  • 日志级别改成 error（因为打印日志会耗时）

5）堆内存优化

-Xms1024m

其他：全量数据获取

nginx 动静分离

静态资源会由客户端发起请求，并且请求是 gulimall.com/static/**，这个请求会重新经过Nginx，因此：

  1. 所有项目的静态资源都应该放在Nginx里面

  2. 规则：[原后端项目中的]/static/*** 所有请求都由 Nginx 直接返回：

    1. 在 nginx 新建文件夹：/mydata/nginx/html/static，将 product 模块的static下的index文件夹拷过来。
      之前的静态资源请求路径：[gulimall.com/]index/img/section_second_list_right_img.png
    2. 修改前端项目内部静态资源的请求路径为：[gulimall.com/]**static/**index/img/section_second_list_right_img.png

  3. 修改 nginx 配置，conf.d/gulimall.conf，监听gulimall.com:80/static，返回 root：
